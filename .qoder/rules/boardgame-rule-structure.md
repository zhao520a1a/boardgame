# 桌游规则结构化整理规范

> 定位：控制桌游规则文档"怎么组织"——从原始素材预处理到最终结构化输出的全流程规范。
> 目标：让任意来源（讲解视频字幕、口述记录、说明书等）的桌游规则经整理后，结构一致、AI可解析、人类可速查。

## 适用范围

当任务涉及**将桌游规则整理为结构化文档**时启用本规则，典型素材来源：

| 素材类型 | 特征 | 常见挑战 |
|---------|------|---------|
| 讲解视频字幕/逐字稿 | 口语化、有示例演示、信息非线性 | 噪声多、视觉指代、规则散落在示例中 |
| 口述/播客记录 | 对话体、有互动讨论 | 信息碎片化、有离题内容 |
| 说明书扫描/OCR | 书面语、结构相对完整 | OCR错字、排版信息丢失 |
| 已有规则文档 | 半结构化 | 格式不统一、缺少字段 |

---

## 一、素材预处理

> 在进入结构化整理之前，先对原始素材做预处理。此步骤是将低质量输入转为可用信息的关键。

### 1.1 素材分类与标记

通读原始文本，将每段内容标记为以下类别之一：

| 类别 | 定义 | 处理方式 |
|------|------|---------|
| **规则陈述** | 直接描述游戏机制、流程、条件的语句 | 保留，归入对应模块 |
| **示例演示** | 通过具体场景/角色名演示规则（如"小红打出守卫…"） | 提取其中隐含的规则，示例本身可丢弃 |
| **视觉指代** | 引用画面内容（如"此图显示的""在这里""像这样"） | 尝试从上下文推断实际内容；无法推断时通过搜索补全 |
| **引导/过渡语** | 讲解者的串场语（如"让我教你""接下来看"） | 丢弃 |
| **推广/互动语** | 非规则内容（如"快收藏""点赞关注""试试这款游戏吧"） | 丢弃 |
| **重复描述** | 同一规则的多次表述（讲解视频常反复强调） | 合并为一次，取最完整的表述 |

### 1.2 噪声过滤模式

以下模式在讲解视频字幕中高频出现，整理时应识别并过滤：

**直接丢弃：**
- 开场白/结束语："大家好""让我N分钟教会你""记得点赞收藏"
- 主观评价："超好玩""这款游戏太经典了""非常推荐"
- 观众互动："评论区告诉我""你们觉得呢"
- 自我指代转换：将"让我教你怎么玩"转为直接陈述规则

**谨慎处理（可能包含规则信息）：**
- 经验总结："其实角色牌上写的已经很好理解了" → 暗示牌面有效果文字，但不含具体规则
- 模糊指代："按照提示去试试" → 可能指向帮助卡内容

### 1.3 从示例中提取规则

讲解视频的核心特征：**规则常嵌套在示例演示中**，而非直接陈述。

提取方法：

1. **识别示例边界**：通过人名（小红、小兰）、"像此时""比如""假设"等标志词定位示例段落
2. **剥离具体角色**：将"小红选择一名玩家猜他的手牌"泛化为"打出者选择一名目标玩家，猜测其手牌"
3. **抽象为规则**：去掉特定情境，保留机制本质
4. **回填条件分支**：示例通常只演示一种结果，需从上下文补全其他分支（如"猜对→出局"和"猜错→无事发生"）

### 1.4 视觉指代解析

视频字幕中常出现指向画面的表述，按可解析程度分级处理：

| 指代类型 | 示例 | 处理方式 |
|---------|------|---------|
| 可从上下文推断 | "此图显示的对应数量" → 上下文提到"钟情标记" | 结合语境补全为具体描述 |
| 可从游戏常识推断 | "所有角色效果在这里" → 标准角色牌效果 | 通过网络搜索官方规则补全 |
| 无法推断 | "这张卡长这样" → 纯视觉内容 | 通过网络搜索官方规则补全 |

### 1.5 字幕错误修正

语音转文字和OCR常产生特定类型的错误：

| 错误类型 | 示例 | 修正方法 |
|---------|------|---------|
| 同音替换 | "气质初见叠牌" → "弃置过间谍的牌" | 结合游戏术语语境修正 |
| 断句错误 | 一句规则被拆成多条字幕 | 依据语义合并为完整句 |
| 吞字/多字 | "出局出局" → "出局" | 去重/补字 |
| 专有名词错误 | 游戏名/角色名识别错误 | 用首次出现的正确写法统一全文 |

> 修正原则：仅修正明显的转录错误，不修改原始规则的实际内容。

### 1.6 信息缺口补全

当原始素材信息不全时（视频字幕尤其常见），**主动通过网络搜索官方规则书补全**，而非留空或标记待补充。补全优先级：

1. 官方规则书 PDF / 出版商网站
2. BGG 等权威桌游数据库
3. 可信的规则讲解网站

## 二、文档总体约定

### 2.1 输出文件结构

每款游戏整理后产出**两个文件**，职责严格分离：

| 文件 | 路径 | 内容 | 原则 |
|------|------|------|------|
| **规则文档** | `{游戏目录}/{游戏中文名}规则.txt` | 纯游戏规则描述 | 只包含玩家需要知道的规则信息，不含任何整理过程痕迹 |
| **整理备注** | `{游戏目录}/整理备注.txt` | 整理过程记录 | 记录规则来源、字幕修正、信息补全来源、版本差异等 |

#### 规则文档中**不得出现**的内容

- `[待补充]`、`[原文: xxx]`、`[推断]` 等过程标记
- 字幕修正记录、待补充清单
- 规则来源说明、信息补全记录
- 版本差异对比（如需说明，放入整理备注）
- 任何"注释""备注""编者按"性质的内容

#### 整理备注文件结构

```markdown
# {游戏中文名}规则整理备注

## 规则来源
- 来源1: {素材类型和简要说明}
- 来源2: ...

## 字幕修正记录
| 原文 | 修正为 | 出现位置 | 修正依据 |
|------|--------|---------|---------|

## 信息补全记录
| 补全项 | 补全内容 | 信息来源 |
|--------|---------|---------|

## 版本差异说明（如有多版本）
| 差异项 | 版本A | 版本B |
|--------|-------|-------|
```

### 2.2 模块标识

每个一级模块使用 **英文ID + 中文名** 双语标题，格式：

```markdown
## {ENGLISH_ID} | {中文名}
```

便于程序化引用（用英文ID）和人类阅读（用中文名）。

### 2.3 内容原则

| 原则 | 说明 |
|------|------|
| 忠实原始 | 保留原始规则的所有信息，不增删规则内容 |
| 纯净规则 | 规则文档只阐述规则，不混入整理过程的注释、标记或待办 |
| 结构优先 | 用表格承载结构化数据，用有序列表承载流程步骤 |
| 消除歧义 | 术语首次出现时给出定义，或在术语表中统一收录 |
| 交叉引用 | 模块间引用使用 `[ENGLISH_ID \| 中文名]` 格式，避免重复描述 |
| 主动补全 | 素材中的信息缺口通过搜索官方规则补全，而非留空标记 |

## 三、标准模块清单

以下为模块的**标准顺序**。标记为 `必填` 的模块每份规则文档必须包含；标记为 `按需` 的模块根据游戏实际情况决定是否纳入。

| 序号 | 模块ID | 中文名 | 必填/按需 | 职责 |
|------|--------|--------|-----------|------|
| 1 | META | 游戏元信息 | 必填 | 游戏身份、类型、人数、核心机制等 |
| 2 | COMPONENTS | 游戏组件 | 必填 | 列举所有实体组件及其用途 |
| 3 | SETUP | 游戏准备 | 必填 | 从开箱到可以开始游戏的全部步骤 |
| 4 | TURN_ORDER | 回合顺序 | 按需 | 起始玩家、行动方向、跳过规则 |
| 5 | TURN_FLOW | 回合流程 | 必填 | 单个回合内的步骤序列 |
| 6 | ACTIONS | 可选行动 | 按需 | 回合中可供选择的行动类型及细则 |
| 7 | SPECIAL_RULES | 特殊规则 | 按需 | 不属于常规流程的独立机制 |
| 8 | ROUND_END | 单轮结束 | 按需 | 单轮结束的触发条件与结算方式 |
| 9 | ROUND_RESET | 轮次交替 | 按需 | 轮次间的重置与保留规则 |
| 10 | SCORING | 计分规则 | 按需 | 得分方式、排名、平局处理 |
| 11 | GAME_END | 游戏终局 | 必填 | 终局触发条件与最终胜利判定 |
| 12 | CARDS | 卡牌效果 | 按需 | 卡牌类游戏的逐张效果说明 |
| 13 | TILES | 板块/地块 | 按需 | 板块放置类游戏的板块说明 |
| 14 | VARIANTS | 变体规则 | 按需 | 人数变体、进阶规则、可选模块 |
| 15 | GLOSSARY | 术语表 | 按需 | 游戏特有术语的统一定义；若术语已在正文中充分简化则可省略 |

## 四、各模块字段规范

### 4.1 META | 游戏元信息

使用表格，字段如下：

| 字段 | 必填 | 说明 | 示例 |
|------|------|------|------|
| 游戏名称 | 是 | 中文名 (英文名) | 情书 (Love Letter) |
| 游戏类型 | 是 | 用 `/` 分隔多个标签 | 卡牌 / 推理 / 淘汰 |
| 玩家人数 | 是 | 范围格式 | 2~4人 |
| 建议人数 | 否 | 最佳体验人数 | 3~4人（最佳4人） |
| 游戏时长 | 否 | 大致时长 | 约20分钟 |
| 适合年龄 | 否 | 最低年龄 | 8岁以上 |
| 一句话介绍 | 否 | 用最通俗的语言概括核心玩法，让人30秒理解游戏 | 每人手里只有1张牌，抽1张打1张，用能力猜测和淘汰对手 |
| 核心机制 | 是 | 用顿号分隔 | 手牌管理、身份推理、玩家淘汰 |
| 游戏结构 | 是 | 单局制/多轮制/实时制等 | 多轮制（每轮独立结算，跨轮累计积分） |
| 主题背景 | 是 | 1~2句话概括 | 扮演公主的追求者，秘密护送情书 |

> 注意：`规则来源` 等整理过程信息不放入 META，归入整理备注文件。

### 4.2 COMPONENTS | 游戏组件

使用表格，每行一个组件：

| 列 | 说明 |
|----|------|
| 组件名称 | 组件的通用名 |
| 数量 | 具体数字或"每人X个" |
| 说明 | 该组件在游戏中的作用 |

### 4.3 SETUP | 游戏准备

- 使用**有序列表**，步骤必须按执行顺序排列
- 步骤中的关键约束用**加粗**标注
- 人数变体规则放在子标题 `### 人数变体 | {具体场景}` 下

### 4.4 TURN_ORDER | 回合顺序

使用表格区分不同场景下的起始玩家规则，并说明行动方向和跳过规则。

### 4.5 TURN_FLOW | 回合流程

- 使用 `### 步骤N: {步骤名}` 格式逐步展开
- 每个步骤包含：动作描述、约束条件、结果
- 步骤间严格按执行顺序排列
- 补充规则（如展示规则、出局处理）放在 `### 补充规则` 下

### 4.6 CARDS / TILES | 卡牌效果 / 板块说明

每张卡牌/板块使用**三级标题 + 表格**描述，标准字段：

**卡牌类：**

| 字段 | 说明 |
|------|------|
| 点数/等级 | 卡牌的数值 |
| 数量 | 牌组中该卡牌的张数 |
| 效果类型 | 攻击/防御/查看/交换/特殊等 |
| 目标 | 效果的作用对象 |
| 执行 | 具体执行的操作 |
| 限制 | 使用时的约束条件 |
| 结果 | 各种情况下的结果分支 |
| 触发时机 | 打出时/弃牌时/持有时等 |

> 按需裁剪字段：不是每张卡都需要所有字段，无关字段可省略。

**板块类：**

| 字段 | 说明 |
|------|------|
| 板块名称 | 板块标识 |
| 正面/反面 | 双面板块的两面内容 |
| 放置规则 | 合法放置的约束条件 |
| 效果 | 放置后触发的效果 |

### 4.7 SCORING | 计分规则

- 得分方式用表格列出
- 平局处理单独说明
- 涉及排名的用表格展示排名与分数的对应关系

### 4.8 GAME_END | 游戏终局

必须包含两部分：
1. **终局触发**：明确列出所有可能的终局触发条件
2. **最终胜利**：胜利判定规则，含平局处理

### 4.9 VARIANTS | 变体规则

每个变体使用三级标题，格式：

```markdown
### {变体名称}（{适用条件}）
```

如 `### 二人局规则（2人游戏时）`

### 4.10 GLOSSARY | 术语表

使用表格，两列：术语 + 定义。收录标准：
- 游戏特有的概念（如"钟情标记""狩猎"）
- 在规则中有特定含义的通用词（如"出局""手牌"）
- 容易产生歧义的表述

## 五、整理流程

当收到原始规则文本时，按以下步骤执行：

### 阶段A: 素材预处理（参见第一章）

1. **判断素材类型**：确认是视频字幕、口述记录、说明书还是其他来源
2. **逐段分类标记**：将每段内容标记为规则陈述/示例演示/视觉指代/噪声
3. **过滤噪声**：移除引导语、推广语、重复描述
4. **示例提取**：从具体示例中抽象出通用规则
5. **指代解析**：尽可能将视觉指代转为具体描述
6. **错误修正**：修正明显的转录/OCR错误
7. **信息补全**：对缺失的关键规则信息，通过网络搜索官方规则书补全

### 阶段B: 结构化整理

8. **识别核心循环**：从预处理后的文本中识别游戏的核心循环（准备→行动→结算→终局）
9. **提取元信息**：填充 META 模块
10. **拆分模块**：将内容按标准模块清单归类，确定哪些 `按需` 模块需要纳入
11. **结构化填充**：按各模块字段规范填写
12. **消除冗余**：重复描述的内容只保留一处，其余改为交叉引用
13. **补充术语表**：回顾全文，将游戏特有术语收录到 GLOSSARY

### 阶段C: 输出分离

14. **清洁规则文档**：确保规则文档中不含任何过程标记（`[待补充]`、`[原文]`、`[推断]`）、注释、待办项或来源说明
15. **生成整理备注**：将字幕修正记录、信息补全来源、版本差异等过程信息写入整理备注文件

### 阶段D: 校验

16. **规则文档校验**：逐项检查校验清单
17. **备注文件校验**：确认所有过程信息已迁移至备注文件

## 六、完整性校验清单

### 规则文档校验

- [ ] 所有 `必填` 模块均已包含
- [ ] META 中核心字段（名称、类型、人数、机制、结构、背景）完整
- [ ] SETUP 步骤可按顺序无歧义执行
- [ ] TURN_FLOW 步骤顺序明确，无遗漏
- [ ] GAME_END 包含终局触发和胜利判定
- [ ] 原始素材中的所有规则信息均已覆盖，无遗漏
- [ ] 未凭推测添加原始素材中不存在的规则
- [ ] 术语表覆盖所有游戏特有概念
- [ ] 模块间无重复描述，交叉引用正确
- [ ] 人数变体（如有）独立标注
- [ ] 文档中**不含**任何 `[待补充]`、`[原文: xxx]`、`[推断]` 标记
- [ ] 文档中**不含**字幕修正记录、待补充清单、来源说明等非规则内容
- [ ] 从示例中提取的规则已泛化，不含特定角色名/场景

### 整理备注校验

- [ ] 规则来源已记录
- [ ] 所有字幕修正均已记录（原文、修正后、位置、依据）
- [ ] 所有网络搜索补全的信息已记录来源
- [ ] 版本差异（如有）已说明

## 七、新手友好写作规范

> 规则文档的最终读者是**没有桌游经验的新手**。以下规范确保文档通俗易懂、快速上手。

### 7.1 术语简化

将专业/生僻术语替换为日常用语，降低认知门槛：

| 简化方向 | 示例 | 说明 |
|---------|------|------|
| 游戏术语口语化 | "亲近值" → "点数" | 选择最直觉的词，一看就懂 |
| 组件名称直觉化 | "好感指示物" → "爱心标记" | 用形象化的词替代抽象名称 |
| 场所名称简化 | "抽牌库" → "牌堆" | 去掉不必要的修饰 |

**原则**：如果一个术语需要解释才能理解，就应该换成不需要解释的词。术语简化到位后，GLOSSARY 模块可省略。

### 7.2 去除繁杂人名

- 角色卡只保留**职称/身份**（如"卫兵""国王""公主"），去除角色专属人名（如"奥德特""艾诺德四世"）
- 人名不影响游戏机制理解，去除后减少新手的记忆负担
- 卡牌标题格式：`角色名（点数N × M张）— 功能关键词`

### 7.3 模块合并

当独立模块内容较少（少于3行）时，合并到相关模块以减少文档碎片化：

| 可合并的模块 | 合并到 | 条件 |
|------------|--------|------|
| TURN_ORDER | TURN_FLOW | 回合顺序规则简单（仅起始玩家+方向） |
| ROUND_RESET | ROUND_END | 轮次重置仅需1~2句话描述 |
| SPECIAL_RULES | CARDS 各卡牌详细说明 | 特殊规则与特定卡牌强绑定 |
| GLOSSARY | 省略 | 术语已在正文中充分简化 |

**原则**：模块合并是为了减少新手在文档间跳转的次数，但不能丢失规则信息。

### 7.4 卡牌展示优化

卡牌类游戏的效果说明采用**总览表 + 详细说明**双层结构：

**第一层：总览表**

一张表速览所有卡牌核心信息，让新手快速建立整体印象：

```markdown
| 点数 | 角色 | 张数 | 能力一句话 |
|------|------|------|-----------|
| 1 | 卫兵 | 5张 | 猜一个人的牌，猜中他出局 |
| ...  | ...  | ...  | ...       |
```

**第二层：详细说明**

每张卡牌用**编号 + 口语化描述**展开，替代每张一个表格的重复格式：

```markdown
**① 卫兵（点数1 × 5张）— 猜牌**
- 选一个人，猜他手里是什么角色（**不能猜卫兵**）
- 猜对 → 对方出局；猜错 → 无事发生
```

**格式要点**：
- 标题包含：编号、角色名、点数×张数、功能关键词
- 正文用无序列表，每条一个操作或规则
- 特殊交互直接写在对应卡牌下方，不单独成章

### 7.5 语言风格

| 要求 | 说明 | 示例 |
|------|------|------|
| 疑问式标题 | 章节标题用新手视角的疑问句，降低距离感 | "一轮怎么结束" 替代 "单轮结束" |
| 概括先行 | 每个流程模块开头用一句话概括 | "每个人的回合只做三件事" |
| 口语化表达 | 用说话的方式写规则，避免书面体 | "谁都不能看" 替代 "不可查看" |
| 强调记忆点 | 对容易忽略的关键规则加提示性短语 | 公主卡备注"要死死攥住！" |
| 避免交叉引用 | 新手不习惯跳转阅读，尽量把信息写在需要的位置 | 特殊规则直接写在卡牌说明中 |
