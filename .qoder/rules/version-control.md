# 版本控制规则 - 文件夹版本管理

## 适用范围

本规则适用于所有 quest 任务的执行过程。每次执行任务时，必须通过创建独立版本文件夹的方式进行版本控制，确保历史版本可追溯、不可覆盖。

## 文件夹命名规范

采用 `{版本号}_{月日}` 格式：

```
v{N}_{MMDD}
```

- **版本号**：从 `v1` 开始递增（v1, v2, v3...）
- **日期**：两位月份 + 两位日期，无分隔符
- 示例：`v1_0211`（第1版，2月11日）、`v2_0315`（第2版，3月15日）

### 版本号确定规则

1. 检查当前工作目录下已有的版本文件夹
2. 取最大版本号 +1 作为新版本号
3. 如果不存在任何版本文件夹，则从 `v1` 开始

## 执行流程

每次 quest 任务启动时，必须按以下步骤执行：

### 步骤 1：扫描已有版本

```
ls 当前工作目录，查找所有符合 v{N}_{MMDD} 格式的文件夹
```

### 步骤 2：创建新版本文件夹

```
mkdir v{新版本号}_{当天月日}
```

### 步骤 3：在版本文件夹内执行任务

所有任务产出的文件和资源都必须保存在该版本文件夹内，包括但不限于：

- 生成的图片文件
- 文案文件
- 配置文件
- 其他任务产出物

### 步骤 4：生成目录文件

任务完成后（或每次新增文件后），在版本文件夹内自动生成或更新 `catalog.md` 文件。

## 目录文件规范（catalog.md）

每个版本文件夹内必须包含 `catalog.md`，内容格式如下：

```markdown
# {版本号} 目录 - {YYYY-MM-DD}

## 版本信息
- 版本号：{版本号}
- 创建时间：{YYYY-MM-DD}
- 任务描述：{本次任务的简要说明}

## 目录结构

​```
v{N}_{MMDD}/
├── catalog.md          # 目录文件
├── file1.png           # 文件说明
├── file2.md            # 文件说明
└── ...
​```

## 文件说明

| 文件名 | 类型 | 说明 |
|--------|------|------|
| catalog.md | 目录 | 本文件，记录文件夹结构 |
| ... | ... | ... |
```

## 强制约束

1. **不可覆盖**：绝对不能修改或覆盖已有版本文件夹中的内容。如需修订，必须创建新版本。
2. **完整性**：每个版本文件夹必须包含该次任务的所有相关文件，不得将文件散落在版本文件夹外。
3. **目录同步**：`catalog.md` 必须在任务结束前更新，准确反映文件夹的最终结构。
4. **独立性**：每个版本文件夹应自包含，不依赖其他版本文件夹中的文件。
