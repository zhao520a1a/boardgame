# .qoder 目录 README 自动维护

> 职责：监测 `.qoder` 文件夹中各子文件夹及文件的变动，自动创建并持续维护每个文件夹内的 `README.md` 文件，确保项目结构文档始终与实际内容保持一致。

## 自动触发条件

当以下任一情况发生时，**自动执行本 Skill**：

| 触发事件 | 说明 |
|---------|------|
| 文件创建 | `.qoder` 下任意子文件夹中新增了文件 |
| 文件修改 | `.qoder` 下任意文件内容发生实质性变更（功能描述、结构调整等） |
| 文件删除 | `.qoder` 下任意文件被移除 |
| 文件夹变动 | 新增或删除了子文件夹 |
| 手动触发 | 用户输入 `/qoder-readme-maintainer` 或要求"更新 .qoder 目录文档" |
| 定期检查 | 当用户开启新会话时，主动检查 README 是否需要更新 |

**判断逻辑**：在执行任何涉及 `.qoder` 目录下文件的操作后（创建、编辑、删除 skill/rule/agent/repowiki），自动扫描变更所在文件夹，对比现有 README.md 内容与实际文件状态，若不一致则触发更新。

## 扫描范围

扫描 `.qoder` 下的所有一级子文件夹：

```
.qoder/
├── agents/       → 生成/更新 agents/README.md
├── rules/        → 生成/更新 rules/README.md
├── skills/       → 生成/更新 skills/README.md（同时更新 skills/Readme.md 概览）
└── repowiki/     → 生成/更新 repowiki/README.md
```

同时维护 `.qoder/README.md` 作为顶层索引。

## README.md 生成规范

### 一、顶层索引（`.qoder/README.md`）

```markdown
# .qoder 配置目录

> 本文件由 qoder-readme-maintainer 自动维护，最后更新：{更新时间}

## 目录结构

| 文件夹 | 用途 | 文件数 | 最近更新 |
|--------|------|--------|---------|
| agents/ | AI Agent 定义 | {数量} | {时间} |
| rules/ | 规则与规范 | {数量} | {时间} |
| skills/ | Agent Skills | {数量} | {时间} |
| repowiki/ | 项目知识库 | {数量} | {时间} |

详见各子文件夹内的 README.md。
```

### 二、子文件夹 README 格式

每个子文件夹的 `README.md` 遵循统一格式：

```markdown
# {文件夹名称}

> 本文件由 qoder-readme-maintainer 自动维护，最后更新：{YYYY-MM-DD}
> 维护状态：✅ 已同步 | ⚠️ 需要审核 | ❌ 待更新

## 概述

{该文件夹的整体功能描述，1-2句}

## 文件清单

| 文件名 | 功能描述 | 关键特性 | 依赖关系 | 维护状态 |
|--------|---------|---------|---------|---------|
| {文件名} | {简洁的功能描述} | {核心特性关键词} | {依赖的其他文件，无则"-"} | {✅/⚠️/❌} |

## 依赖关系图（如适用）

{用简洁文字描述文件间的依赖和协作关系}

## 变更记录

| 日期 | 变更类型 | 文件 | 说明 |
|------|---------|------|------|
| {YYYY-MM-DD} | 新增/修改/删除 | {文件名} | {简要说明} |
```

### 三、文件功能描述生成规则

为每个文件生成摘要时，必须遵循以下规则：

1. **读取文件内容**：完整读取文件，理解其核心功能
2. **提取关键信息**：
   - 对于 Skill 文件（SKILL.md）：提取 `name`、`description` 和核心职责
   - 对于 Rule 文件（.md）：提取规则标题和适用范围
   - 对于 Agent 文件（.md）：提取 Agent 名称和调度逻辑
   - 对于 Repowiki 文件：提取知识点标题和内容范围
3. **功能描述要求**：
   - 不超过 30 字
   - 使用动词开头（如"定义…"、"规范…"、"生成…"）
   - 突出核心用途，避免泛泛描述
4. **关键特性标注**：用 2-4 个关键词概括，用 `、` 分隔
5. **依赖关系标注**：
   - 明确标注被其他文件引用或引用其他文件的关系
   - 格式：`→ 被{文件名}引用` 或 `← 依赖{文件名}`

## 维护状态判定规则

| 状态 | 符号 | 判定条件 |
|------|------|---------|
| 已同步 | ✅ | README 中的描述与文件实际内容一致 |
| 需要审核 | ⚠️ | 文件内容发生变更但 README 尚未确认更新 |
| 待更新 | ❌ | 文件已删除但 README 仍有记录，或文件新增但 README 尚无记录 |

## 执行流程

### 自动触发流程

```
检测到 .qoder 文件变动
    ↓
确定变动所在文件夹（agents/rules/skills/repowiki）
    ↓
扫描该文件夹内所有文件
    ↓
读取现有 README.md（如果存在）
    ↓
对比文件清单差异
    ├─ 新增文件 → 读取内容，生成功能摘要，追加到文件清单
    ├─ 删除文件 → 从文件清单移除，记录到变更记录
    └─ 修改文件 → 重新读取内容，更新功能摘要，标记为 ⚠️
    ↓
更新 README.md
    ↓
更新 .qoder/README.md 顶层索引
    ↓
输出变更摘要通知用户
```

### 手动触发流程

用户输入以下任一命令触发：
- `/qoder-readme-maintainer` — 全量扫描所有子文件夹
- `/qoder-readme-maintainer agents` — 仅扫描 agents 文件夹
- `/qoder-readme-maintainer rules` — 仅扫描 rules 文件夹
- `/qoder-readme-maintainer skills` — 仅扫描 skills 文件夹
- `/qoder-readme-maintainer repowiki` — 仅扫描 repowiki 文件夹

### 定期检查流程

在新会话开始时：
1. 检查各文件夹的文件数量是否与 README 中记录的一致
2. 对比文件名列表是否有差异
3. 若发现不一致，主动提示用户并建议更新

## 质量保障机制

### 准确性验证

每次更新 README 时执行以下检查：

1. **文件存在性检查**：README 中列出的每个文件确实存在于文件夹中
2. **文件完整性检查**：文件夹中的每个文件都在 README 中有记录
3. **描述一致性检查**：重新读取文件内容，验证功能描述是否仍然准确
4. **格式完整性检查**：确保 README 格式符合模板规范，无缺失字段

### 变更检测策略

- **内容级检测**：对比文件的核心功能描述是否发生变化
- **结构级检测**：对比文件数量、文件名是否发生变化
- **标记过期**：当文件修改后但 README 未更新时，自动标记 ⚠️

## 输出规范

每次更新完成后，向用户输出简洁的变更通知：

```
📋 .qoder 目录文档已更新

变更范围：{文件夹名}
├── ✅ 新增：{文件名} — {功能描述}
├── 📝 更新：{文件名} — {变更说明}
└── 🗑️ 移除：{文件名}

已更新文件：
- {文件夹}/README.md
- .qoder/README.md
```

## 注意事项

1. **不干扰用户工作流**：自动触发时静默执行，仅在完成后输出简洁通知
2. **保留用户手动编辑**：如果 README 中有用户手动添加的内容（非自动生成部分），更新时保留
3. **最小化变更**：仅更新有变动的部分，不重写整个文件
4. **变更记录保留**：变更记录按时间倒序排列，最多保留最近 20 条
5. **编码一致性**：所有 README 使用 UTF-8 编码
6. **skills/Readme.md 联动**：更新 skills/README.md 时，同步检查并更新 skills/Readme.md 中的架构概览信息
